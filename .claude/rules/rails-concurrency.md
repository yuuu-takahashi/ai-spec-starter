# 並行性・データ整合性規約

## Race Condition の防止

- 同時更新が起こり得る処理（在庫、残高、ステータス遷移等）を特定する
- 読み取りから更新までのウィンドウを最小化する

## ロック戦略

- **楽観的ロック**: `lock_version` カラムを使い、`ActiveRecord::StaleObjectError` を処理する
- **悲観的ロック**: `with_lock { }` で行ロックを取得。ロック範囲は最小限にする
- 複数行をロックする場合、常に同じ順序でロックする（デッドロック回避）
- ロック中に外部 API 呼び出しやファイル I/O を行わない

## ジョブの冪等性

- 同じジョブが複数回実行されても結果が同じになるように設計する
- `find_or_create_by` + DB ユニーク制約で重複作成を防止する
- 外部 API 呼び出しには `idempotency_key` を使う

## トランザクション

- 関連する複数操作は同一トランザクション内で実行する
- トランザクション内に外部 I/O を含めない
- 副作用（メール、通知）は `after_commit` でトランザクション確定後に実行する

## DB 制約

- 必須カラムには `NOT NULL` 制約を設定する
- ユニークであるべきデータには DB レベルの `UNIQUE` 制約を設定する（モデルの `validates uniqueness` だけでは不十分）
- 外部キー制約で参照整合性を保証する

## DO NOT

- `validates :uniqueness` だけで一意性を保証しない（DB 制約を併用する）
- トランザクション内で長時間処理を行わない
- ロック順序を統一せずに複数リソースをロックしない
