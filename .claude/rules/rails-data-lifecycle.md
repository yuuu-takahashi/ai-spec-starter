# データライフサイクル規約

## 削除方式を明確にする

- 論理削除・物理削除・匿名化のどれを使うかをモデルごとに決める
- `destroy` が論理削除か物理削除か曖昧にしない。メソッド名で明示する
- 論理削除を使う場合、標準クエリで削除済みレコードが自動除外されることを確認する

## 論理削除（Soft Delete）

- `deleted_at` カラムで管理する
- `default_scope` での除外は避ける。明示的な scope（`active`, `with_deleted`）を使う
- 論理削除したレコードの関連データ（`has_many`）の扱いを決めておく
- 復元が必要なケースを想定し、復元手順を用意する

## データ保持ポリシー

- 各データ型の保持期間をビジネス・法的要件に基づいて定義する
- 保持期限超過データの定期削除は非同期ジョブで実行する
- 物理削除前にキャッシュ（Redis 等）・検索インデックスからも削除する

## 監査ログ

- 重要なデータの CREATE / UPDATE / DELETE を監査ログに記録する
- 変更者（who）・変更内容（what）・変更日時（when）を含める
- 監査ログの記録は `after_commit` で行う（トランザクション確定後）

## PII・匿名化

- 個人情報の削除要求に対応できる匿名化処理を用意する
- 匿名化は `update_columns` で実行する（バリデーション・コールバック回避）
- API レスポンスに不要な個人情報を含めない

## DO NOT

- 削除方式を決めずに `destroy` を使わない
- 監査が必要なデータを監査ログなしで変更しない
- 論理削除レコードを `default_scope` で隠さない（クエリ漏れの原因）
- 物理削除時にキャッシュ・検索インデックスの整合性を忘れない
