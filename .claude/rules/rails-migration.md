# マイグレーション運用規約

## 基本原則

- 1 マイグレーション = 1 つの変更。無関係な変更を混ぜない
- マイグレーション名は変更内容を明確に（`AddEmailToUsers`, `RemoveDeprecatedColumn`）
- `rails db:rollback` で戻せることを確認する

## 段階移行（破壊的変更）

カラム名変更・型変更・削除は段階的に行う:

1. **追加**: 新カラム/テーブルを追加
2. **両対応**: 新旧両方に対応するコードをデプロイ
3. **切替**: 新方式に完全移行
4. **削除**: 旧カラム/テーブルを削除

## 大規模テーブル（50 万行以上）

- インデックス追加は `algorithm: :concurrently`（PostgreSQL）を使う
- NOT NULL 制約は段階移行で追加する（NULL 許可 → データ埋め → NOT NULL 化）
- 大量データ更新はマイグレーション内で同期実行せず、非同期ジョブに委譲する

## データ操作の安全性

- `update_columns` を使う（バリデーション・コールバック回避）
- Service Object やモデルのビジネスロジックを呼び出さない
- 可能なら SQL 直接実行（`execute "UPDATE ..."`）で完結させる

## インデックス

- `where` / `JOIN` / `ORDER BY` 対象カラムにインデックスを設定する
- 外部キーには必ずインデックスを付ける
- 削除前に使用クエリがないことを確認する

## DO NOT

- マイグレーション内で `User.find_each { |u| u.save! }` しない（バリデーション発火）
- `change` だけで可逆性が不明確な場合は `up` / `down` を明示する
- 本番想定のデータ量でテストせずにデプロイしない
