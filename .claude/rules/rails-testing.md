# RSpec テスト規約

## 基本原則

- プロダクションコードと同じ PR にテストを含める。テストを後回しにしない
- 仕様が不明確な場合は仕様化テスト（characterization test）で先に挙動を確認する
- テストコードはドキュメントとして機能させる。構造から仕様が読み取れること

## テストケース名

- `describe` / `context` / `it` の名前だけで仕様が読み取れるようにする
- `context` は条件（「〜の場合」）、`it` は期待結果（「〜すること」）
- 曖昧な名前（`works`, `success`, `correctly`）は禁止。具体的な振る舞いを書く

```ruby
# ❌ 仕様が読み取れない
it 'works' do ...

# ✅ 仕様が読み取れる
context 'ユーザーが未認証の場合' do
  it '401 を返すこと' do ...
end
```

## 構造

- ネストは 3 レベル以下に抑える
- Arrange（準備）→ Act（実行）→ Assert（検証）を空行で区切る
- `it` の中では実行と検証のみ。データ準備は `before` / `let!` で行う

## テストデータ

- テストデータは `before` または `let!` で準備する。`it` 内でデータを作らない
- `let!` を使う。遅延評価の `let` は避ける
- テストデータはトップレベルで共通化しない。各 `describe`/`context` 内で作成する
- `build` と `create` を使い分ける。DB アクセス不要なら `build` / `build_stubbed`
- Factory の `trait` で状態のバリエーションを表現する

## モック・スタブ

- モックは最小限に。外部 API のみモックする
- 自前メソッドを直接モックしない。WebMock で HTTP レベルでモックする
- 時間依存は `travel_to` / `freeze_time` で固定する

## カバレッジ

- 正常系・異常系・境界値を網羅する
- ビジネスロジック（Service / Model）は 90% 以上、金銭・在庫計算は 100%
- `expect` は 1 テスト 1〜2 個。曖昧な名前（`works`, `success`）は避ける

## DO NOT

- `subject` を使わない。テスト対象は `it` 内で明示的に呼び出す
- `before(:all)` を使わない（テスト間の独立性が壊れる）
- テスト間でインスタンス変数を共有しない
- `it` 内でテストデータを作成しない（`before` / `let!` で準備する）
- スタイル修正と機能テストを同じ PR に混ぜない
