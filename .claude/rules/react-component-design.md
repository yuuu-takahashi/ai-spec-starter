# コンポーネント設計規約

## 目的

UI 実装で責務が混在することを防ぎ、変更しやすく・読みやすく・レビューしやすい構成を保つ。

- 変更理由が異なる関心事を同じ場所に混ぜない
- 業務ルール・状態管理・表示責務の境界を明確にする
- props と依存から責務が読み取れる状態を保つ
- 分離そのものを目的にせず、見通しと保守性を優先する

ディレクトリや依存の大枠は `react-directory-design.md` に従う。

---

## 基本原則

- 1 コンポーネント / 1 hook には、可能なら主な変更理由を 1 つにする
- 業務ルールは上位で閉じ、下位には判定結果だけ渡す
- 依存は `Page → Section → Feature Display → Primitive/UI` の一方向のみ
- UI のための軽い整形は表示層に置いてよい
- 分離は手段。責務が明確なら 1 ファイルに留めてよい

---

## 4 層の責務分離

依存方向は次に限定する。下位が上位（画面専用コード等）を参照しない。

`Page → Section → Feature Display → Primitive/UI`

### 1. Page 層

ページ全体の成立条件を扱う。

#### 責務

- ルーティング依存（パラメータ・クエリの解決）
- データ取得の起点（API / loader / store 呼び出しの開始）
- 権限・プラン・リダイレクト条件など業務条件の判定
- Section に渡す入力の組み立て

#### やらないこと

- 細かい表示の組み立てばかり持つ
- Primitive に近い見た目制御を抱える
- 末端向けの軽い整形を過剰にここへ寄せる

### 2. Section 層

画面内の意味単位（例: プロフィール塊、請求塊、一覧塊）を構成する。

#### Section の責務

- 複数 hooks / state の組み合わせ
- 表示可否・空状態・ローディングなど画面内の判断
- Feature Display に渡す値の整形（生レスポンスをそのまま流さない）
- モーダル開閉・タブ切替など、その画面セクション内の UI 制御

#### Section でやらないこと

- ルーティングに直接依存する（ページに寄せる）
- 生の API レスポンスを末端 UI にそのまま渡す
- 画面外でも再利用すべき責務まで抱える

### 3. Feature Display 層

業務文脈に閉じた表示のまとまり（例: ユーザーカード、請求サマリ）。

#### Feature Display の責務

- 受け取った値で表示を構成する
- 軽い表示整形（氏名連結、日付・金額フォーマット、`statusLabel` など）
- 条件に応じた表示切り替え（**渡された結果**に基づく）

#### Feature Display でやらないこと

- データ取得
- ルーティング・画面遷移の責務
- `role` / `plan` 等を使った業務条件の**自分での**判定

### 4. Primitive / UI 層

見た目と基本挙動のみの汎用 UI（Button, Input, Modal 等）。

#### Primitive の責務

- variant / size / disabled など見た目と基本挙動
- セマンティック HTML とキーボード・フォーカス対応
- 汎用コンポーネントとしての再利用性

#### Primitive でやらないこと

- 業務ルール・業務語彙に依存した命名や props
- `role`, `plan`, `organizationStatus` などを直接受け取る

---

## 責務の分け方

### 業務ルールは上位で閉じる

Page / Section で判定し、下位には材料ではなく結果を渡す。

- 許容する渡し方の例: `canEdit`, `shouldShowBilling`, `isDisabled`, `statusLabel`
- Primitive には業務判定の材料（`role`, `permissions` 生データ等）を渡さない

### 軽い整形 vs 業務判断

**Feature Display でよい例**: 氏名連結、フォーマット、状態に応じた**文言**（材料は既に絞られている前提）

**上位で扱う例**: プラン・権限・フローに基づく表示制御、操作可否の決定

「軽い整形」の境界はチームでブレやすい。迷ったら業務判断は Page / Section に寄せる。

### hook / ファイル分割

hook や別ファイルへ切り出すのは次のいずれかに当てはまるとき。

- 再利用したい
- UI なしでテストしたい
- 1 ファイルの見通しが悪い、または変更理由が複数混ざっている

次の程度はコンポーネント内に留めてよい。

- 数行で閉じる derived value
- その画面だけの短いハンドラ
- UI と密接な小さな分岐

## Props 設計

### 必要最小限

大きなオブジェクトを丸ごと渡さず、そのコンポーネントが実際に使う値に絞る。

```tsx
// 避けたい
<UserCard user={user} />

// より良い
<UserCard name={user.name} email={user.email} avatarUrl={user.avatarUrl} />
```

### props 名は「そのコンポーネントが知ってよい関心事」だけを表す

props はデータの通り道ではなく**公開インターフェース**である。上位の画面事情や業務判定の材料を、名前とともに下位へ漏らしすぎない。

- 判断**材料**より判断**結果**を渡すことが多い
- props を見て責務が読めるようにする
- 「この props を見て、ここまで業務判断をしているように見えるか？」が Yes なら漏れている

```tsx
// 悪い寄り（下位が業務文脈を知りすぎる）
<SubmitButton role={role} plan={plan} organization={organization} />

// 良い寄り（ボタンが知ってよいことだけ）
<SubmitButton disabled={!canSubmit} loading={isSubmitting} />
```

- Primitive: 汎用的な props
- Feature Display: 業務文脈に沿った名前だが、末端ほど「結果」に寄せる

---

## Hook 設計

- hook は state・イベント・**UI 非依存**の導出値を返す
- 見た目を決めすぎない。避ける例: `badgeColor`, `buttonVariant`, `iconName`, `className` を hook が返してコンポーネントがただ描くだけ、にしない
- 「何が起きているか」は hook、「どう見せるか」はコンポーネント側で決める

---

## State 設計

- 置き場所は「責務に最も自然な階層」: ローカル → 上位へ引き上げ → アプリ全体で意味があるものだけグローバル
- 同一情報を複数 state で持たない
- 計算で求められる値は state にしない（derived state）。例: `filteredItems`, `isEmpty`, `fullName`（元から導出可能な場合）

---

## 副作用・ライフサイクル

- `useEffect` は外部同期・購読・タイマー・リスナー等の**副作用**に限定する
- props / state から同期的に計算できる値を effect で別 state にコピーしない
- クリーンアップ必須の処理は必ず解放する
- レンダー中に副作用や高コスト処理を走らせない
- 依存配列は意図を持って正確に。`eslint-plugin-react-hooks` に従う

---

## アクセシビリティ

- セマンティック HTML（ボタンやリンクを `div` で代用しない）
- キーボード操作可能を前提にする
- 必要なところでフォーカス管理する
- ARIA は不足なく、過剰にもしない
- 見た目だけで意味を伝えない

---

## 迷ったときの確認

次に Yes で答えられるか。

- 主な変更理由は 1 つか
- 業務ルールは Page / Section で閉じているか
- 下位が画面事情・業務条件を知らずに済むか
- props から責務が読めるか
- UI 変更で業務ロジックまで追わなくて済むか / 業務変更で末端 UI を大量に触らなくて済むか
- 分離で見通しが悪くなっていないか

---

## DO NOT

- 1 ファイルに取得・権限判定・状態管理・描画を詰め込む
- Primitive / UI に `role` や `plan` 等の業務条件を渡す
- props drilling 回避のためだけに雑なグローバル state を増やす
- 導出可能な値を state に持つ
- `useEffect` で同期可能な値を state に逃がす
- 分離目的だけで hook / ファイルを細かく増やす
- hook が見た目まで決定する
- API レスポンスの生データを末端 UI にそのまま流す
- 下位から上位の画面専用コードを参照する・循環依存にする

---

## レビュー観点（PR）

- このファイルの責務は主に 1 つか
- 業務ルールは Page / Section で閉じているか
- Feature Display / Primitive が業務条件を知らずに済むか
- props が過剰でないか・大きなオブジェクトを丸ごと渡していないか
- props 名が「知ってよい関心事」に収まっているか
- derived state や state の重複がないか
- `useEffect` が本当に副作用だけか
- hook が見た目を決めすぎていないか
- 依存方向が崩れていないか
- 分離の結果、追いづらくなっていないか
