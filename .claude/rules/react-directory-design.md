# ディレクトリ設計規約

## 基本方針

- 「ルート（`app/`）」「feature」「shared」の境界を明確に固定する
- ルーティング構造とアプリの責務分割を一致させる
- どのページでも同じ探索パスで意図した実装に辿り着ける状態を保つ
- 「この仕様変更ならどのファイルを見るか見当がつく」を設計の評価基準にする

## 機能単位の整理

- 機能（feature）やドメイン単位でコードをまとめる
- UI / ロジック / データ取得の配置を一貫させる
- ページ固有のコードを共有レイヤーに漏らさない
- 機能を丸ごと削除するとき、関連ファイルをまとめて整理・削除しやすい構造にする

## 依存の方向

- 依存は一方向に保つ: `app/` → `features/` → `shared/`
- 下位モジュールが上位の画面専用コードを参照しない
- `shared/` は特定の feature に依存しない

## コロケーション

- コンポーネント・hooks・styles・tests を近接配置する
- 共有対象と機能固有の区別を明確にする
- 一貫性を優先してコロケーションルールを統一する

## 命名・公開 API

- フォルダ / ファイル名の規則を一貫させる
- barrel export（`index.ts`）は公開面を狭く保てる場合だけ使う
- 深すぎる相対 import を避ける（パスエイリアスの活用）

## DO NOT

- `components/` に何でも集めない
- 同じ feature を `app/`・`features/`・`lib/` に分散させない
- 命名規則を画面ごとにブレさせない
- 循環依存やレイヤーを跨ぐ逆依存を作らない
- 特定の構成パターン（Atomic Design 等）をそのまま正解としない。プロジェクトに合わせる
