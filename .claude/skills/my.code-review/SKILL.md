# Code Review Skill — Google Engineering Practices ベース

## 最上位原則

> CL が完璧でなくても、その変更がシステムのコードの全体的な健康状態を改善すると確実にわかれば、レビュアーは CL を積極的に承認すべき。

完璧さの追求ではなく **継続的な改善** を求める。開発者の進捗と品質確保のバランスを常に意識する。

## 入力パラメータ

- **`$1`**: PR 番号、ファイルパス、または未指定
- **`$2`**: 重点観点（`設計`, `security` など）— 指定時はその観点を重点的にレビュー

## Workflow

### Step 1: レビュー対象を特定する

| 条件                        | 方法                                  |
| --------------------------- | ------------------------------------- |
| PR 番号あり                 | `gh pr diff <number>` で差分取得      |
| PR 番号なし（現在ブランチ） | `gh pr diff` で差分取得               |
| ローカル変更のみ            | `git diff` または `git diff --staged` |
| 特定ファイル指定            | Read ツールで直接読む                 |

PR の場合は追加情報も取得する:

```bash
gh pr view <number> --json title,body,headRefName,baseRefName,files,additions,deletions,changedFiles
```

差分が大きい場合（500 行超）は、変更ファイル一覧を先に確認し、影響範囲を把握してからファイル単位で読む。

### Step 2: 11 観点でレビューする

以下の観点を確認する。全てにコメントする必要はなく、問題がある箇所にだけ指摘を行う。
変更規模が小さい場合は関連する観点のみで十分。重点観点が指定されている場合はその観点を優先的にレビューする。

#### 並行レビュー（推奨）

差分が大きい場合（100 行超）や観点を網羅したい場合は、Agent ツールで 3 グループを並行実行する。
各エージェントには差分全文（または `gh pr diff` の結果）とレビュー対象の情報を渡す。

| グループ  | 観点                                           | 焦点                             |
| --------- | ---------------------------------------------- | -------------------------------- |
| A: 構造   | 設計・機能性・複雑性・コンテキスト             | コードは正しく設計されているか   |
| B: 品質   | テスト・パフォーマンス・セキュリティ           | コードは安全で効率的か           |
| C: 可読性 | 命名・コメント・スタイル・ドキュメンテーション | コードは読みやすく保守しやすいか |

**エージェントへの指示テンプレート:**

```text
以下の差分をコードレビューしてください。担当観点: [観点リスト]

レビュー対象: [PR タイトル or コミットメッセージ]
差分:
[差分テキスト]

各指摘について以下を含めてください:
- コメントラベル: [must], [imo], [nits], [ask], [fyi], [nr]
- ファイルパス:行番号
- なぜ問題か
- 修正前/修正後のコード（該当する場合）

問題がない観点は「問題なし」と明記。最後に担当観点のサマリーテーブル（✅/⚠️/❌ + 問題数）を出力。
```

3 エージェントの結果を受け取ったら、Step 3 でマージしてレポートを構成する。

差分が小さい場合（100 行以下）は並行化のオーバーヘッドが大きいため、単一で順にレビューする。

#### 1. 設計 (Design)

- CL 全体の設計がシステムに適合しているか
- コードは適切な場所に配置されているか（ファイル・モジュール・パッケージ）
- 既存のコードと重複する実装がないか
- 今このタイミングで追加すべき機能か

#### 2. 機能性 (Functionality)

- 開発者の意図通りに動作するか
- エッジケース、境界値、並行処理の問題はないか
- UI 変更がある場合、ユーザー体験に問題はないか

#### 3. 複雑性 (Complexity)

- コードを読んですぐに理解できるか
- 「賢すぎる」実装になっていないか
- オーバーエンジニアリングの兆候はないか（推測に基づく将来の拡張、不要な抽象化）
- 関数やクラスの責務が大きすぎないか

#### 4. テスト (Tests)

- 適切なテスト（ユニット・結合・E2E）が含まれているか
- テストはコード破壊時に正しく失敗するか
- 各テストは簡潔で有用なアサーションを使っているか
- テスト自体が保守しやすいか

#### 5. 命名 (Naming)

- 変数・関数・クラス・ファイル名が役割を明確に伝えているか
- 名前は十分に説明的でありながら、長すぎないか

#### 6. コメント (Comments)

- コメントは「なぜ（Why）」を説明しているか（「何（What）」ではなく）
- コードが自明でない箇所に適切な説明があるか
- コードをシンプルにすることで不要にできるコメントはないか

#### 7. スタイル (Style)

- プロジェクトのスタイルガイド・リンター設定に準拠しているか
- 既存コードとの一貫性が保たれているか

#### 8. ドキュメンテーション (Documentation)

- 新しい API や設定項目に対するドキュメント更新があるか
- 破壊的変更がある場合、マイグレーションガイドがあるか

#### 9. パフォーマンス (Performance)

- N+1 クエリ、不要なループ、メモリリークの可能性はないか
- キャッシュ戦略は適切か
- 大量データの処理でスケーラビリティに問題はないか

#### 10. セキュリティ (Security)

- インジェクション（SQL、XSS、コマンド）の可能性はないか
- 認証・認可は適切に実装されているか
- 機密情報（トークン、パスワード、API キー）がハードコードされていないか
- 入力値のバリデーションは十分か

#### 11. コンテキスト (Context)

- PR の目的・背景が明確か
- 変更がプロジェクトの方針・ロードマップと整合しているか
- 他の進行中の変更との衝突はないか

### Step 3: レビューコメントを構成する

#### コメントの原則（Google プラクティスより）

1. **コードについてコメントし、人についてコメントしない**
   - NG: 「なぜスレッドを使ったのですか？」
   - OK: 「この並行処理モデルは複雑さに対してパフォーマンスメリットが見合わない可能性があります」

2. **「なぜ」を説明する** — 指摘の根拠や品質への影響を説明する

3. **指示と質問のバランス** — 問題を指摘しつつ、開発者に修正方法を考える余地を残す。明確な改善案がある場合は具体的なコードで示す

4. **肯定的フィードバック** — 優れた実装を見つけたら積極的に伝える（メンタリング効果）

#### コメントラベル

| ラベル   | 意味             | ブロッキング |
| -------- | ---------------- | ------------ |
| `[must]` | 必須修正         | Yes          |
| `[imo]`  | 提案・個人的見解 | No           |
| `[nits]` | 細かい改善提案   | No           |
| `[ask]`  | 質問             | No           |
| `[fyi]`  | 参考情報         | No           |
| `[nr]`   | お手すきで       | No           |

#### 指摘の書き方

指摘ごとに以下を含める:

- **ファイルパス:行番号**
- **なぜ問題か**: 問題の理由と影響の説明
- **修正前のコード**: 実際のコードをコードブロックで（プレースホルダー不可）
- **修正後のコード**: 改善案をコードブロックで具体的に

### Step 4: レビューレポートを出力する（二重出力: 必須）

レビューレポートは **必ず以下の両方** に出力する。片方だけでは不可。

1. **ターミナル出力（会話）**: レポート全文をそのまま会話上に表示する。ユーザーがすぐに内容を確認できるようにする。
2. **ファイル保存**: `.claude/skills/paths.md` の **`REVIEWS`** キー（`${KNOWLEDGE_BASE}/records/reviews/`）に Markdown として保存する（詳細は「ファイル保存」セクション参照）。保存完了後、ファイルパスを会話内に表示する。

この 2 つは同時に行い、どちらかを省略してはならない。

## ファイル保存

保存先は `.claude/skills/paths.md` の **`REVIEWS`** キーに従う（`${KNOWLEDGE_BASE}/records/reviews/`）。

レビュー結果は必ず上記ディレクトリに Markdown ファイルとして保存する。
ディレクトリが存在しない場合は作成する。

**ファイル名の規則:**

- PR の場合: `pr-<number>-<短い説明>.md`（例: `pr-42-add-auth-middleware.md`）
- ローカル差分の場合: `<YYYY-MM-DD>-<短い説明>.md`（例: `2026-04-05-refactor-api-handler.md`）
- 同名ファイルが既にある場合はサフィックス `-v2`, `-v3` を付ける

保存後、ファイルパスを会話内に表示する。

## PR への投稿（任意）

PR に直接コメントを投稿する場合は、ユーザーに確認してから `gh pr review` を使う:

```bash
gh pr review <number> --comment --body "レビューコメント"
gh pr review <number> --approve --body "LGTM コメント"
gh pr review <number> --request-changes --body "変更要求コメント"
```

投稿前に必ずユーザーの確認を取ること。

## レビューの心構え

### 承認の判断基準

CL を承認すべき状態:

- 完璧ではなくても、コードベースの全体的な健康状態を改善している
- 重要な設計上の問題がない
- 機能的に正しく動作する
- テストが適切に含まれている

CL を差し戻すべき状態:

- 設計に根本的な問題がある
- セキュリティ上の脆弱性がある
- 既存機能を壊すリグレッションがある
- テストが不足している

### コンフリクト発生時

1. 技術的事実とデータに基づいて議論する（個人的好みではなく）
2. 複数の有効な選択肢がある場合は、作成者の選好を尊重する
3. コードベースの一貫性を優先する
4. 解決しない場合は、チーム内でのエスカレーションを提案する

### CL が大きすぎる場合

大きな CL はレビューが難しく品質も下がる。以下を提案する:

- 機能単位での分割
- リファクタリングと機能追加の分離
- テスト追加を別 CL にする

ただし、既に出された CL に対しては、分割を強制せず建設的にレビューを進める。

言語別スキルが存在しない場合は、標準の 11 観点でレビューを実施する。
