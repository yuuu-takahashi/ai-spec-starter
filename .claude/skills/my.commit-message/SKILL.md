# my:commit

## 概要

変更内容からコミットメッセージを自動生成するスキルです。

## いつ使うか

- コミットメッセージの書き方に迷ったとき
- 適切なメッセージを素早く生成したい
- コミット規則に従ったメッセージを作成したい

## 使用方法

```text
/my:commit
```

実行時にステージされた変更を分析します。

## メッセージフォーマット

```text
<prefix>: <理由>なため、<変更内容>
```

## Prefix の種類

| Prefix     | 説明                       | 例                                                      |
| ---------- | -------------------------- | ------------------------------------------------------- |
| `feat`     | 新しい機能                 | `feat: 使いやすさを向上させるため、ユーザー認証を実装`  |
| `fix`      | バグ修正                   | `fix: エラーが発生するため、nullチェックを追加`         |
| `docs`     | ドキュメント変更           | `docs: セットアップを簡単にするため、README を更新`     |
| `style`    | コードスタイル変更         | `style: コード品質を向上させるため、フォーマットを統一` |
| `refactor` | リファクタリング           | `refactor: 可読性を向上させるため、関数を分割`          |
| `perf`     | パフォーマンス向上         | `perf: ロード時間を短縮するため、キャッシュを追加`      |
| `test`     | テスト関連                 | `test: カバレッジを向上させるため、テストケースを追加`  |
| `chore`    | ビルド・ツール・ライブラリ | `chore: 開発効率を向上させるため、axiosを追加`          |

## 処理フロー

### 1. 変更内容の確認

- `git status` で変更ファイルを確認
- `git diff --cached` で変更内容を確認
- 変更の種類（追加・修正・削除）を把握

### 2. Prefix の決定

変更内容から適切な prefix を選択：

- 新機能追加 → `feat`
- バグ修正 → `fix`
- ドキュメント変更 → `docs`
- コードスタイル変更 → `style`
- リファクタリング → `refactor`
- パフォーマンス改善 → `perf`
- テスト追加・修正 → `test`
- ビルド設定・依存関係 → `chore`

### 3. 理由と変更内容の抽出

- **理由（なぜ）**: 変更の目的・背景
  - 例: 「ユーザー認証を実装するため」「エラー処理を改善するため」
- **変更内容（何を）**: 具体的な変更
  - 例: 「ログイン機能を追加」「例外処理を追加」

### 4. メッセージ生成

形式: `<prefix>: <理由>なため、<変更内容>`

### 5. 複数変更がある場合

複数の変更が含まれる場合：

- 主要な変更を1つ選ぶか、コミットを分割することを推奨
- 1つのコミット = 1つの目的

## 出力例

変更内容から自動生成されたメッセージが提示されます：

```text
推奨メッセージ:
feat: ユーザー認証を実装するため、ログイン機能を追加

変更内容:
- src/auth/login.ts (新規作成)
- src/pages/login.tsx (新規作成)
- package.json (bcrypt を追加)
```

## 注意事項

- 理由が書きにくい場合でも後で見てわかるメッセージに
- 日本語で記述（日本人チーム向け）
- 過去形ではなく、現在形・名詞形で記述（「追加」「実装」など）
- 簡潔で明確なメッセージを心がける
- 60字程度の簡潔さを推奨
