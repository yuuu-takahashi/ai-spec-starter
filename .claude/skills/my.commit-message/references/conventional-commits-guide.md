# Conventional Commits 形式ガイド

コミットメッセージの統一形式。変更内容を明確に伝え、自動化ツール (変更ログ生成、バージョン管理など) と連携可能な形式です。

## 基本形式

```text
<prefix>: <理由>なため、<変更内容>
```

### 各要素の役割

| 要素     | 説明                              | 例                                 |
| -------- | --------------------------------- | ---------------------------------- |
| prefix   | 変更の種類 (feat, fix, docs など) | `feat`                             |
| 理由     | 変更の目的・背景                  | 「ユーザー利便性を向上させるため」 |
| なため   | 接続詞（日本語固有）              | 「なため」                         |
| 変更内容 | 具体的な変更内容                  | 「ログイン機能を追加」             |

### 例

```text
feat: ユーザーの利便性を向上させるため、ログイン機能を追加

fix: エラーが発生するため、nullチェックを追加

docs: セットアップを簡単にするため、README を更新
```

## Prefix（接頭辞）の種類

### feat（機能追加）

**説明**: 新しい機能を追加

**使用場面**:

- 新しい API を実装
- ユーザーインターフェースに新機能を追加
- データモデルを拡張

**例**:

```text
feat: ユーザー体験を向上させるため、ダークモードを実装

feat: アナリティクスを強化するため、ページビューをトラッキング

feat: 生産性向上のため、キーボードショートカットを追加
```

**コミット後の影響**:

- メジャーバージョン (MAJOR) への影響なし
- マイナーバージョン (MINOR) をインクリメント
- 変更ログの「Features」セクションに自動記載

### fix（バグ修正）

**説明**: バグを修正

**使用場面**:

- エラー処理の追加
- ロジックの誤りを修正
- パフォーマンス問題の解決

**例**:

```text
fix: ページロード時にクラッシュするため、メモリリークを修正

fix: フォーム送信に失敗するため、バリデーションロジックを修正

fix: 日本語入力が不正に処理されるため、エンコーディング処理を修正
```

**コミット後の影響**:

- メジャーバージョン (MAJOR) への影響なし
- パッチバージョン (PATCH) をインクリメント
- 変更ログの「Bug Fixes」セクションに自動記載

### docs（ドキュメント変更）

**説明**: ドキュメントのみの変更 (コード変更なし)

**使用場面**:

- README ファイルの更新
- コードコメントの追加・修正
- API ドキュメントの更新
- インストール手順の記載

**例**:

```text
docs: セットアップ手順を明確にするため、README を更新

docs: API の使い方をわかりやすくするため、JSDoc コメントを追加

docs: 環境構築を簡単にするため、Docker 設定を説明
```

**コミット後の影響**:

- バージョンに影響しない（docs のみ）
- 自動リリースの対象外
- 変更ログには含まれない

### style（スタイル変更）

**説明**: コードのスタイル・フォーマットの変更 (機能変更なし)

**使用場面**:

- コードフォーマッターの適用 (Prettier, autopep8 など)
- インデントの統一
- セミコロンの追加・削除
- スペースの統一

**例**:

```text
style: コード一貫性を確保するため、フォーマットを統一

style: 可読性を向上させるため、インデント幅を4に統一

style: Prettier を適用するため、コードをリフォーマット
```

**コミット後の影響**:

- バージョンに影響しない
- 機能変更がないため、自動リリース対象外
- 変更ログには含まれない

### refactor（リファクタリング）

**説明**: コード構造を改善 (機能変更なし)

**使用場面**:

- 関数・クラスの分割
- 変数名の改名
- 重複コードの統合
- デザインパターンの改善

**例**:

```text
refactor: メンテナンス性を向上させるため、複雑な関数を分割

refactor: 可読性を高めるため、変数名を意味のある名前に変更

refactor: コード重複を削減するため、共通ロジックを抽出
```

**コミット後の影響**:

- 機能に変更がないため、バージョンに影響しない
- 自動リリース対象外
- 内部品質向上の記録として重要

### perf（パフォーマンス向上）

**説明**: パフォーマンスを改善

**使用場面**:

- キャッシング機構の追加
- アルゴリズムの最適化
- データベースクエリの改善
- バンドルサイズの削減

**例**:

```text
perf: ページロード時間を短縮するため、画像を遅延ロード化

perf: データベースアクセスを高速化するため、インデックスを追加

perf: パッケージサイズを削減するため、不要な依存をリムーブ
```

**コミット後の影響**:

- 機能向上として扱われることもある
- パッチバージョン (PATCH) をインクリメント
- パフォーマンス関連のコミットとして記録

### test（テスト追加・修正）

**説明**: テストコードのみの変更

**使用場面**:

- ユニットテストの追加
- 統合テストの追加
- テストケースの修正
- テスト覆率の向上

**例**:

```text
test: ロジックの信頼性を確保するため、ユニットテストを追加

test: エッジケースに対応するため、テストケースを拡充

test: 既存テストが失敗するため、テスト条件を修正
```

**コミット後の影響**:

- バージョンに影響しない
- コード品質向上として記録

### chore（ツール・ライブラリ変更）

**説明**: ビルド設定、依存関係、開発ツールなどの変更

**使用場面**:

- npm パッケージの更新
- webpack・Babel などのビルドツール設定変更
- CI/CD パイプラインの設定
- 開発環境の構成変更

**例**:

```text
chore: 開発効率を向上させるため、axios を依存関係に追加

chore: ビルドプロセスを簡素化するため、webpack 設定を更新

chore: 開発環境をセットアップするため、docker-compose を導入
```

**コミット後の影響**:

- バージョンに影響しない
- 自動リリース対象外

## メッセージの書き方ガイド

### 1. 「理由」の部分

**良い例** (変更の目的・背景が明確):

```text
ユーザー体験を向上させるため
エラーが発生するため
パフォーマンスを改善するため
メンテナンス性を高めるため
セキュリティを強化するため
```

**避けるべき例** (漠然としている):

```text
feat: 何かを追加         × (何を？)
feat: UI を改善        × (どう改善？)
feat: コードを改善      × (具体性がない)
```

### 2. 「変更内容」の部分

**良い例** (具体的な変更が明確):

```text
ログイン機能を追加
nullチェックを追加
README を更新
関数を分割
キャッシュを導入
```

**避けるべき例** (抽象的):

```text
色々変更          × (何を？)
いろいろ修正      × (具体性がない)
アップデート       × (抽象的)
```

### 3. 文字数と簡潔さ

**推奨**: 60字程度

**計算例**:

```text
feat: ユーザー認証を実装するため、ログイン機能を追加
     (前半20字) (中盤14字) (後半17字) = 計51字 ✓

feat: システム全体の複雑性を軽減し、メンテナンス性を大幅に向上させるため、複数の小さな機能に分割
     (長すぎる - 推奨範囲を超える) ✗
```

**短縮テクニック**:

```text
元: 「複数のログイン方法を対応させるため」
短: 「複数ログイン方式に対応するため」

元: 「ユーザーが簡単に使えるようにするため」
短: 「使いやすさを向上させるため」
```

### 4. 時制

**推奨**: 現在形・名詞形（命令形でない）

```text
✓ 「ログイン機能を追加」    (名詞形・現在形)
✓ 「ログイン機能追加」      (名詞形)
✓ 「Add login feature」    (英語: 命令形)

✗ 「ログイン機能を追加した」  (過去形)
✗ 「ログイン機能を追加します」 (丁寧形)
✗ 「ログイン機能を追加する」   (現在形動詞)
```

## 複数の変更がある場合

### パターン1: 1つのコミットに複数変更を含める

```text
feat: ユーザー体験を向上させるため、認証機能を実装

以下の変更を含む:
- ログイン機能を追加
- パスワードリセット機能を追加
- ユーザープロフィール管理を追加
```

注意: 推奨されない。できるだけ複数コミットに分割。

### パターン2: コミットを分割（推奨）

コミット1:

```text
feat: ユーザー認証を実装するため、ログイン機能を追加
```

コミット2:

```text
feat: パスワード回復を可能にするため、リセット機能を追加
```

コミット3:

```text
feat: ユーザー情報管理を強化するため、プロフィール機能を追加
```

**メリット**:

- 各コミットが単一の目的を持つ
- git bisect で問題を特定しやすい
- マージやリベースが容易
- 変更ログが見やすい

## コミットメッセージの実例

### 例1: 新機能 (feat)

```text
feat: ユーザー利便性を向上させるため、ダークモード機能を実装
```

### 例2: バグ修正 (fix)

```text
fix: ページクラッシュを解決するため、メモリリークを修正
```

### 例3: リファクタリング (refactor)

```text
refactor: メンテナンス性を高めるため、複雑な関数を小分け関数に分割
```

### 例4: パフォーマンス向上 (perf)

```text
perf: 初期ロード時間を短縮するため、画像を遅延ロード化
```

### 例5: ドキュメント更新 (docs)

```text
docs: セットアップ手順を明確にするため、README を充実
```

### 例6: テスト追加 (test)

```text
test: バグを防ぐため、エッジケースのテストを追加
```

### 例7: 依存ライブラリ更新 (chore)

```text
chore: 開発環境を整備するため、axios を 1.4.0 にアップグレード
```

## Breaking Changes（破壊的変更）

API や公開インターフェースに互換性がない変更の場合。

### 記述方法

```text
feat: 複数キーワード検索を実現するため、search API を再設計

BREAKING CHANGE: search 関数のシグネチャが変更されました。
新シグネチャ: search(options: SearchOptions) => Promise<Result[]>
旧シグネチャ: search(query: string, limit: number) => Result[]

移行ガイド: options オブジェクトを使用してください
```

### 影響

- メジャーバージョン (MAJOR) をインクリメント
- 変更ログに「Breaking Changes」として記載
- ユーザーへの通知が必須

## ツール連携

### 自動変更ログ生成

```bash
# conventional-changelog を使用
npx conventional-changelog -i CHANGELOG.md -s
```

生成される変更ログ:

```markdown
## [1.2.0](https://github.com/user/repo/compare/v1.1.0...v1.2.0)

### Features

- ユーザー認証を実装するため、ログイン機能を追加
- ユーザー体験を向上させるため、ダークモード機能を実装

### Bug Fixes

- ページクラッシュを解決するため、メモリリークを修正

### Performance Improvements

- 初期ロード時間を短縮するため、画像を遅延ロード化
```

### 自動バージョン管理

```bash
# semantic-release を使用
npx semantic-release
```

自動的にバージョンをインクリメント:

- feat → マイナーバージョン (1.2.0 → 1.3.0)
- fix → パッチバージョン (1.2.0 → 1.2.1)
- BREAKING CHANGE → メジャーバージョン (1.2.0 → 2.0.0)

## チェックリスト

コミットメッセージ作成時の確認項目:

- [ ] 適切な prefix を選択している (feat, fix, docs など)
- [ ] 理由が明確に記述されている
- [ ] 変更内容が具体的である
- [ ] 60字程度の簡潔さが保たれている
- [ ] 現在形・名詞形で記述している
- [ ] 日本語で統一している
- [ ] 複数の変更が含まれる場合、分割を検討している
- [ ] Breaking changes がある場合、BREAKING CHANGE を記載している
