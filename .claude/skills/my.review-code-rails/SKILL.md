# Rails / RSpec レビュー

このスキルは、Rails レビューで毎回ぶれやすい観点を揃えるための入口です。
実装規約を全部ここに抱え込むのではなく、最初に全体像をつかみ、必要な補助資料へ進むための案内役に寄せています。

## 使う場面

- `.rb` ファイルのレビュー
- バックエンド PR のレビュー
- Rails の設計やベストプラクティス確認

## 読む順番

最初から全部読む必要はありません。まずこのファイルで地図を掴み、差分に応じて必要な資料だけ読む想定です。

1. 差分から変更の中心を決める
2. 下の「レビュー観点」から該当する perspective を読む
3. Rails 固有のパターンがあれば pattern を読む
4. `guide_04_output_format.md` の形式で findings を出す
5. 迷ったら公式ドキュメントで検証する

## レビュー観点（perspective）

6 つの観点を優先度順に並べています。全部見る必要はなく、変更内容に関連するものだけで十分です。

### Must（問題があればマージ前に修正）

| #   | 観点             | 問い                                        | ファイル                     |
| --- | ---------------- | ------------------------------------------- | ---------------------------- |
| 1   | **設計**         | 責務分離は適切か？変更しやすい構造か？      | `perspective_01_design.md`   |
| 2   | **セキュリティ** | SQLi, XSS, CSRF, 認証・認可の漏れはないか？ | `perspective_02_security.md` |

### Should（強く推奨）

| #   | 観点               | 問い                                       | ファイル                        |
| --- | ------------------ | ------------------------------------------ | ------------------------------- |
| 3   | **パフォーマンス** | N+1, メモリ, 不要なクエリはないか？        | `perspective_03_performance.md` |
| 4   | **テスト**         | 壊れたら気づけるか？テストは信頼できるか？ | `perspective_04_testing.md`     |

### Supporting（補助・一貫性）

| #   | 観点       | 問い                                             | ファイル                       |
| --- | ---------- | ------------------------------------------------ | ------------------------------ |
| 5   | **命名**   | Rails の命名慣習に沿っているか？意図が伝わるか？ | `perspective_05_naming.md`     |
| 6   | **複雑性** | 認知負荷が高すぎないか？次の人が読めるか？       | `perspective_06_complexity.md` |

スタイル（インデント、フォーマット）は RuboCop に委譲する。レビューで指摘しない。

## Rails 固有パターン（pattern）

特定の実装パターンに遭遇したとき、「何が壊れやすいか」を深掘りしたファイル群です。perspective が横串なら、pattern は縦串。

| 対象                   | ファイル                      |
| ---------------------- | ----------------------------- |
| ドメイン駆動設計       | `pattern_06_ddd.md`           |
| パフォーマンス分析手法 | `pattern_08_perf_analysis.md` |

## 補助ファイルの使い分け

レビュー対象に応じて、次のファイルだけ読めば十分です。

| シーン                       | 読むファイル                                                   |
| ---------------------------- | -------------------------------------------------------------- |
| 初めて使う                   | `guide_01_quickstart.md`                                       |
| レビューフローを確認         | `guide_02_workflow.md`                                         |
| パターン別の読み先を知りたい | `guide_03_patterns.md`                                         |
| 出力形式を確認               | `guide_04_output_format.md`                                    |
| レビュー出力の例             | `examples/sample-review.md`                                    |
| 複雑な AR クエリ             | `perspective_03_performance.md`                                |
| トランザクション管理         | `perspective_02_security.md`                                   |
| RSpec テスト設計             | `perspective_04_testing.md` + `perspective_06_complexity.md`   |
| セキュリティ脆弱性           | `perspective_02_security.md`                                   |

## 判断原則

- 既存リポジトリの方針を優先する
- 自動で直る style より、壊れる挙動や性能劣化を先に指摘する
- 指摘は file/line と再現条件を付ける
- Rails らしい書き方より、プロダクトの一貫性と安全性を優先する
- 最新仕様や曖昧な挙動はローカル資料より公式ドキュメントを優先して確認する
