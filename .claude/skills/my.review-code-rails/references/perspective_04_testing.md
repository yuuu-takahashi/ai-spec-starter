## テスト

レビュー時のテスト品質確認チェックリスト。
実装規約の詳細は `.claude/rules/rails-testing.md` を参照。

### 最重要: 仕様とテストの関係

- [ ] **仕様の理解**: テストを書く前に、対象の仕様が明確になっているか
- [ ] **仕様化テスト**: 仕様が不明確な場合、characterization test で現在の挙動を先に確認しているか

> 仕様がわからなければ安全にリファクタリングするためのテストコードを書けない。仕様を理解することが最優先。

### テスト構造

- [ ] **describe/context/it**: 名前だけで仕様が読み取れるか（`context "〜の場合"`, `it "〜すること"`）
- [ ] **曖昧な名前**: `works`, `success`, `correctly` のような曖昧な名前がないか
- [ ] **AAA パターン**: Arrange / Act / Assert が空行で区切られているか
- [ ] **1テスト1検証**: 各テストの `expect` が 1〜2 個以内か

### テストデータ

- [ ] **let! の使用**: `let` ではなく `let!` で事前にデータを作成しているか（遅延評価の罠がないか）
- [ ] **コンテキスト局所化**: テストデータが必要な context 内で定義されているか（トップレベルで共有していないか）
- [ ] **Factory**: Factory が最小限のデータで構成されているか（過剰な trait がないか）

### テスト品質

- [ ] **独立性**: テスト間で状態が共有されていないか（`before(:all)` が使われていないか）
- [ ] **モック制限**: モックは外部 API・時刻など制御不能な依存のみに使われているか
- [ ] **pending**: 未実装テストが `pending` で構造化されているか（空の `it` がないか）

### カバレッジ

- [ ] **ビジネスロジック**: 主要なビジネスロジックのカバレッジが 90% 以上あるか
- [ ] **金融・課金**: 金銭に関わるロジックは 100% カバーされているか
- [ ] **境界値**: エッジケース（nil, 空文字, 最大値, 0件）がテストされているか

### テスト種別ごとの確認

- [ ] **Model spec**: バリデーション、スコープ、コールバックの動作確認があるか
- [ ] **Request spec**: ステータスコード、レスポンスボディ、認証・認可の確認があるか
- [ ] **Service spec**: 正常系、異常系、トランザクションの確認があるか

### perspective でのみ言及する補足観点

以下は `rails-testing.md` でカバーされていない確認事項:

- [ ] **仕様化テストの活用**: あやふやな仕様をテストで分析してからリファクタリングしているか
- [ ] **テスト実行速度**: 不要な DB アクセスやスリープで遅くなっていないか
- [ ] **CI 安定性**: ランダム順序実行で落ちるテストがないか（`--order random`）
