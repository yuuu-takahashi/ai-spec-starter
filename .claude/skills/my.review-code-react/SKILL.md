# Next.js / React レビュー

このスキルは、フロントエンドレビューで毎回ぶれやすい観点を揃えるための入口です。
実装規約を全部ここに抱え込むのではなく、最初に全体像をつかみ、必要な補助資料へ進むための案内役に寄せています。

## 使う場面

- `.ts`, `.tsx`, `.js`, `.jsx` のレビュー
- フロントエンド PR のレビュー
- Next.js の設計やベストプラクティス確認

## 読む順番

最初から全部読む必要はありません。まずこのファイルで地図を掴み、差分に応じて必要な資料だけ読む想定です。

1. 差分から変更の中心を決める
2. 下の「レビュー観点」から該当する perspective を読む
3. `guide_02_output_format.md` の形式で findings を出す
4. 迷ったら公式ドキュメントで検証する

## レビュー観点（perspective）

観点を優先度順に並べています。全部見る必要はなく、変更内容に関連するものだけで十分です。

### Must（問題があればマージ前に修正）

| #   | 観点             | 問い                                                       | ファイル                      |
| --- | ---------------- | ---------------------------------------------------------- | ----------------------------- |
| 1   | **設計**         | コンポーネント責務は適切か？Server/Client 境界は正しいか？ | —                             |
| 2   | **セキュリティ** | XSS, 環境変数漏洩, 認証・認可の漏れはないか？              | —                             |

### Should（強く推奨）

| #   | 観点               | 問い                                                         | ファイル                    |
| --- | ------------------ | ------------------------------------------------------------ | --------------------------- |
| 3   | **パフォーマンス** | 不要な再レンダリング, バンドルサイズ, キャッシュは適切か？   | —                           |
| 4   | **テスト**         | 壊れたら気づけるか？テストは信頼できるか？                   | `perspective_02_testing.md` |

### Supporting（補助・一貫性）

| #   | 観点       | 問い                                             | ファイル |
| --- | ---------- | ------------------------------------------------ | -------- |
| 5   | **命名**   | 慣習に沿っているか？意図が伝わるか？             | —        |
| 6   | **複雑性** | 認知負荷が高すぎないか？次の人が読めるか？       | —        |

スタイル（インデント、フォーマット）は ESLint / Prettier に委譲する。レビューで指摘しない。

## 判断原則

- 既存リポジトリの方針を優先する
- `use client` は必要最小限に留める
- データ取得、キャッシュ、再検証は明示的であるべき
- loading / error / empty state を見落とさない
- A11y と運用コストを性能最適化と同じ優先度で見る
- 自動で直る style より、壊れる挙動や性能劣化を先に指摘する
- 指摘は file/line と再現条件を付ける
- 最新仕様や曖昧な挙動はローカル資料より公式ドキュメントを優先して確認する
