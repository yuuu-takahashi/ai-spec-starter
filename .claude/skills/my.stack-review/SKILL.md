# Stack Review — 技術スタック評価・選定スキル

## 目的

このスキルは「強い技術」を推奨するのではなく、**今の自分/チームにとって最適か** を判断する。
技術の人気度（star 数、企業スポンサー等）は参考情報であり、判断の主軸にはしない。

### 誠実な技術選定の原則

技術選定の理由には2種類ある。これを混ぜてはいけない。

- **積極的な理由（技術起因）**: その技術の特性が要件に合致する。例: リアルタイム性のために GC なし言語を選ぶ、型安全性のために TypeScript を採用する
- **消極的な理由（非技術起因）**: チームの習熟度、学習コスト、採用市場、組織の事情など。技術自体の良し悪しではなく、使う側の都合

消極的な理由で技術を不採用にすること自体は正当。ただし、それを「技術の問題」として語るのは不誠実。
レポートでは、各判断が積極的理由によるものか消極的理由によるものかを明示する。

> 参考: uhyo「技術選定の審美眼」— 消極的な理由を積極的な理由として正当化するのではなく、それぞれを正直に分けて述べることが、長期的に正しい意思決定を導く。

## 入力パラメータ

- **`$1`**: モード指定（`solo` = 個人開発 / `team` = チーム開発 / 未指定 = 自動判定）
- **`$2`**: 対象（`all` = 全棚卸し / ライブラリ名 = 個別評価 / 未指定 = all）

モード未指定の場合は、リポジトリの状態（contributors 数、CODEOWNERS の有無、CI 設定等）から自動判定する。

## Workflow

### Phase 1: 棚卸し（Inventory）

リポジトリ内の依存管理ファイルを読み、使用中の技術スタックを一覧化する。

**読むべきファイル（存在するもののみ）:**

| エコシステム | ファイル |
| --- | --- |
| Node.js | `package.json`, `yarn.lock`, `pnpm-lock.yaml` |
| Ruby | `Gemfile`, `Gemfile.lock` |
| Python | `requirements.txt`, `pyproject.toml`, `Pipfile` |
| Go | `go.mod` |
| Rust | `Cargo.toml` |
| その他 | `.tool-versions`, `Dockerfile`, `docker-compose.yml` |

**出力:** 技術スタック一覧表（カテゴリ別）

```markdown
| カテゴリ | 技術 | バージョン | 用途 |
| --- | --- | --- | --- |
| 言語 | Ruby | 3.3.0 | バックエンド |
| FW | Rails | 7.1 | Web フレームワーク |
| ライブラリ | Sidekiq | 7.2 | 非同期ジョブ |
```

### Phase 2: 4軸評価

各技術を以下の4軸で **分離して** 評価する。軸を混ぜないことが重要。
詳細な評価基準は `references/evaluation-criteria.md` を参照。

#### 軸1: 技術評価（Technical Quality）

その技術が用途に対して筋が良いかを見る。

- API 設計の質と一貫性
- 型安全性（型定義の有無・質）
- テスト容易性（モック・スタブの書きやすさ）
- 拡張性（プラグイン、ミドルウェア等）
- 複雑性の妥当さ（用途に対してオーバーキルでないか）

#### 軸2: 安全性・継続性評価（Safety & Continuity）

継続利用に耐えるかを見る。**これが最も重要な軸。**

- 脆弱性対応の速さと透明性（CVE 対応実績）
- メンテナンス頻度（直近のリリース日、コミット頻度）
- 破壊的変更の頻度と移行ガイドの質
- バス係数（主要メンテナーが1人だけでないか）
- コミュニティの健全性（Issue 対応、PR マージ速度）

#### 軸3: 個人適合評価（Personal Fit）— solo モード重視

- 学習コスト（既存スキルからの距離）
- 自分の技術スタックとの親和性
- 開発速度への寄与（ボイラープレート量、DX）
- ドキュメント・チュートリアルの充実度
- トラブル時に自力解決できるか

#### 軸4: 組織適合評価（Organizational Fit）— team モード重視

- チーム内でレビューできる人数
- 障害発生時に対応できる人数
- 新メンバーのオンボーディング容易性
- 属人化リスク（特定の人しか触れない状態にならないか）
- 採用市場での人材の見つけやすさ

### Phase 3: 判定

各技術に対して以下の3段階で判定する。

| 判定 | 意味 | 条件 |
| --- | --- | --- |
| **継続** | そのまま使い続けてよい | 4軸すべてに重大な問題がない |
| **要検討** | 置換を検討すべき | いずれかの軸で懸念がある |
| **置換推奨** | 積極的に代替を探すべき | 安全性・継続性に問題がある、または複数軸で懸念 |

新規採用の場合は「採用可」「条件付き採用」「非推奨」の3段階。

### Phase 4: 推奨アクション

判定結果に基づき、具体的な次のステップを提示する。

- **継続**: バージョン更新のみ推奨、など
- **要検討**: 代替候補のリストと比較ポイント
- **置換推奨**: 移行計画の骨子（段階的移行 vs 一括置換）

## 評価時の禁止ルール

以下の判断パターンは明示的に禁止する。理由は `references/evaluation-criteria.md` に詳述。

1. **star 数だけで推奨しない** — star は認知度であり品質ではない
2. **contributors 数だけで安全と判断しない** — アクティブかどうかが重要
3. **有名企業の支援だけで信頼しない** — 企業は方針変更でプロジェクトを放棄することがある
4. **「皆が使っている」だけで採用しない** — 自分/チームの文脈での適合性が最優先
5. **最新＝最良と判断しない** — 枯れた技術の安定性も正当に評価する
6. **消極的理由を積極的理由に偽装しない** — 「チームが慣れていない」は正当な消極的理由だが、「この技術は複雑すぎる」にすり替えない。理由の種類を正直に分けて述べる
7. **技術を十分に理解せずに評価しない** — 表面的な印象や伝聞だけで判断せず、実際の API 設計やドキュメントを確認する

## 情報収集の方法

Web 検索で以下を確認する（利用可能な場合）:

- npm / RubyGems / PyPI のダウンロード推移
- GitHub リポジトリの最終コミット日、Issue/PR 対応状況
- 既知の脆弱性（CVE、Snyk、GitHub Advisory）
- 公式ドキュメントの更新頻度
- 主要メンテナーの活動状況

Web 検索が使えない場合は、ローカルの lock ファイルとコード利用状況から推測する。

## モード別の重み付け

### solo モード

- 技術評価: ★★★
- 安全性・継続性: ★★★
- 個人適合: ★★★★★（最重視）
- 組織適合: ★（参考程度）

### team モード

- 技術評価: ★★★
- 安全性・継続性: ★★★★★（最重視）
- 個人適合: ★★（参考程度）
- 組織適合: ★★★★★（最重視）
