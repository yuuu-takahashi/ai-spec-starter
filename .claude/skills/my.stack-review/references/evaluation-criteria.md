## スコア定義

| スコア | 意味 | 基準 |
|--------|------|------|
| **A** | 問題なし | 当該軸において懸念事項がない |
| **B** | 軽微な懸念 | 認識すべき点はあるが、すぐに対処が必要ではない |
| **C** | 要対応 | 明確な問題があり、対処計画が必要 |

## 軸1: 技術評価の詳細基準

### A（問題なし）の条件
- API が一貫しており、直感的に使える
- TypeScript 型定義が公式に提供されている（JS エコシステムの場合）
- テストが書きやすい設計（DI、モック容易性）
- 用途に対して複雑すぎず、シンプルすぎない

### B（軽微な懸念）の条件
- API に一部非一貫な部分がある
- 型定義が DefinitelyTyped 経由（メンテナンスが公式より遅れる可能性）
- テストにやや工夫が必要（グローバル状態への依存等）
- 機能が用途に対してやや過剰だが、不要部分を無視できる

### C（要対応）の条件
- API が頻繁に変わる、または設計に一貫性がない
- 型定義がない、または著しく不正確
- テストが実質的に書けない設計
- 用途に対して著しくオーバーキル、またはアンダーキル

## 軸2: 安全性・継続性の詳細基準

### 確認項目と判断基準

#### メンテナンス活性度
| 指標 | A | B | C |
|------|---|---|---|
| 最終リリース | 6ヶ月以内 | 1年以内 | 1年以上前 |
| 最終コミット | 3ヶ月以内 | 6ヶ月以内 | 6ヶ月以上前 |
| Issue 対応 | 平均2週間以内 | 平均1ヶ月以内 | 放置が目立つ |

※ 「完成した」ライブラリ（標準的なユーティリティ等）は更新頻度が低くても問題ない場合がある。
  機能追加の余地が少なく、バグも枯れている場合は、更新頻度だけで C にしない。

#### 脆弱性対応
| 指標 | A | B | C |
|------|---|---|---|
| CVE 対応 | 迅速（数日〜2週間） | 対応するが遅い | 未対応の CVE がある |
| セキュリティポリシー | SECURITY.md あり | なし but 対応実績あり | なし and 対応実績不明 |

#### バス係数（属人性）
| 指標 | A | B | C |
|------|---|---|---|
| アクティブメンテナー | 3人以上 | 2人 | 1人 |
| 組織バッキング | 複数組織 | 1組織 | 個人のみ |

**重要な注意:** 個人メンテナーだから即座に C ではない。
個人でも長期間安定してメンテナンスしている実績があれば B とする。
逆に、企業バッキングがあっても突然放棄された事例（左記の例: Lerna の一時放棄、Atom の終了）を忘れない。

#### 破壊的変更
| 指標 | A | B | C |
|------|---|---|---|
| メジャーバージョンの頻度 | 2年以上の間隔 | 1〜2年 | 1年未満 |
| 移行ガイド | 充実 | あるが不十分 | なし |
| codemod / 自動移行 | あり | なし | — |

## 軸3: 個人適合の詳細基準

### 評価の観点

1. **学習コスト**
   - 既存スキルからの距離: 使い慣れた言語/パラダイムか？
   - 概念の数: 独自概念がいくつあるか？
   - 「Hello World」から「実用」までの距離

2. **技術スタック親和性**
   - 既存のビルドツール、テストフレームワークとの統合
   - 他の依存との相性（バージョン競合等）
   - エコシステムの方向性との一致

3. **開発速度への寄与**
   - ボイラープレートの量
   - デバッグのしやすさ（エラーメッセージの質）
   - 開発ツール（DevTools、IDE プラグイン等）の充実度

4. **自力解決能力**
   - ドキュメントの質と量
   - ソースコードの読みやすさ（問題発生時に自分で追えるか）
   - Stack Overflow / GitHub Issues での解決事例の豊富さ

## 軸4: 組織適合の詳細基準

### 評価の観点

1. **レビュー体制**
   - チーム内でコードレビューできる人数
   - その技術に関する社内ナレッジの有無
   - コードの可読性（チームメンバーが読めるか）

2. **障害対応**
   - 障害時に調査・修正できる人数
   - ランタイムのデバッグ容易性
   - 監視・ログとの統合のしやすさ

3. **オンボーディング**
   - 新メンバーが戦力になるまでの期間
   - 社内ドキュメント・研修の有無
   - 外部研修・書籍の充実度

4. **属人化リスク**
   - 「あの人しか触れない」状態になるリスク
   - ナレッジ共有の仕組みがあるか
   - 退職・異動時の影響度

5. **採用市場**
   - その技術を扱えるエンジニアの市場規模
   - 求人での訴求力（プラスにもマイナスにもなる）

## 危険信号パターン

以下のパターンを検出した場合、評価を慎重に行う。

### 即座に「要検討」以上にすべきパターン
- 未対応の既知の脆弱性がある
- メンテナーが「もうメンテナンスしない」と宣言している
- 後継プロジェクトへの移行が公式に推奨されている
- ライセンスが最近変更された（特に制限的な方向へ）

### 注意して見るべきパターン
- メンテナーが1人で、活動が減少傾向
- フォーク版の方が活発になっている
- 依存しているライブラリ自体がメンテナンス停止
- ダウンロード数は多いが、新規プロジェクトでの採用が減少傾向

## 積極的理由と消極的理由の分離

技術選定の理由は「積極的（技術起因）」と「消極的（非技術起因）」に分かれる。
レポート内で各判断の根拠を述べる際、どちらの種類の理由かを必ず明示する。

### 積極的な理由（技術起因）の例
- 「リアルタイム性の要件があり、GC の停止時間が許容できないため Rust を選択」
- 「Web API との親和性が高く、型安全性による開発効率改善が見込めるため TypeScript を採用」
- 「データベースの ACID 特性が必要であり、最終整合性モデルでは要件を満たせないため PostgreSQL を選択」

### 消極的な理由（非技術起因）の例
- 「チームに Go 経験者が多いため Go を継続利用」（正当だが、技術的優位性の話ではない）
- 「学習コストが高く、プロジェクトの期限に間に合わないため不採用」
- 「採用市場でのエンジニア確保が困難なため別の技術を選択」

### 混同の禁止パターン

以下は誠実でない記述の例。レポートでこのような書き方をしてはいけない。

| 不誠実な記述 | 何が問題か | 誠実な記述 |
|---|---|---|
| 「Haskell は複雑すぎて実用的でない」 | チームの習熟度の問題を技術の欠陥にすり替えている | 「Haskell は技術的には適しているが、チーム内に経験者がおらず学習期間を確保できない（消極的理由）」 |
| 「Rust はメモリ管理が面倒」 | 所有権システムは安全性のための設計判断であり「面倒」は使う側の主観 | 「Rust の所有権モデルは安全性に寄与するが、チームの現在のスキルセットでは生産性が低下する見込み（消極的理由）」 |
| 「このライブラリは知名度が低いから危険」 | 知名度と安全性は別の軸 | 「コミュニティが小さく情報が少ないため、トラブル時の自力解決コストが高い（消極的理由）。脆弱性対応自体は迅速に行われている（積極的理由で問題なし）」 |

## よくある誤判断パターン

このスキルが陥りがちな判断ミスと、その回避方法。

### 誤: 「star が多い = 良い技術」
**回避:** star は認知度の指標。品質・安全性・適合性とは別軸。
Moment.js は star が多いが、公式にメンテナンスモードであり新規採用は非推奨。

### 誤: 「contributors が多い = 安全」
**回避:** contributors の数ではなく、**アクティブな** contributors の数と分布を見る。
過去に1回だけ PR を出した人はカウントに意味がない。

### 誤: 「Google/Meta/Microsoft が支援 = 安心」
**回避:** 企業は戦略変更でプロジェクトを放棄する。
Angular.js → Angular の移行、Atom の終了、React のライセンス騒動などの事例あり。
企業支援は加点要素ではあるが、それだけで安全性を担保しない。

### 誤: 「新しい = 良い」
**回避:** 新しいライブラリは API が不安定なリスクがある。
枯れた技術の安定性は正当に評価すべき。
ただし「枯れた」と「放棄された」は異なる。

### 誤: 「依存が少ない = 良い」
**回避:** 依存の少なさは一般的にはプラスだが、車輪の再発明のコストとトレードオフ。
依存の質（メンテナンス状況、セキュリティ）こそが重要。
