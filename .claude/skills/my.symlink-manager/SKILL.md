# Symlink Manager — リポジトリ関連のシンボリックリンク管理

現在の作業ディレクトリ（リポジトリ）に関係するシンボリックリンクを管理するスキル。

## 基本方針

- すべての操作でユーザーに確認を取ってから実行する（特に削除・修復）
- 出力は日本語で行う
- レポートは Markdown 形式で生成する

## スコープ: 「リポジトリに関係する」とは

以下の2種類のシンボリックリンクが対象:

1. **リポジトリ内のシンボリックリンク** — リポジトリディレクトリ配下にあるシンボリックリンク
2. **リポジトリを指すシンボリックリンク** — ホームディレクトリ配下にあり、リンク先がリポジトリ内のパスを指しているもの

### リポジトリルートの特定

`git rev-parse --show-toplevel` でリポジトリルートを取得する。取得できない場合は `$PWD` をリポジトリルートとして扱う。

### スキャン方法

```bash
REPO_ROOT=$(git rev-parse --show-toplevel 2>/dev/null || echo "$PWD")

# 1. リポジトリ内のシンボリックリンク
find "$REPO_ROOT" -type l -not -path '*/.git/*' -not -path '*/node_modules/*' \
  -not -path '*/vendor/*' -not -path '*/.bundle/*' 2>/dev/null

# 2. ホームディレクトリからリポジトリを指すシンボリックリンク
#    $HOME の浅い階層（maxdepth 3）をスキャンし、リンク先が $REPO_ROOT を含むものだけ抽出
find "$HOME" -maxdepth 3 -type l 2>/dev/null | while read link; do
  target=$(readlink -f "$link" 2>/dev/null)
  case "$target" in "$REPO_ROOT"*) echo "$link";; esac
done
```

各リンクについて `readlink` でリンク先を取得し、`test -e` でリンク先の存在を確認する。

## 機能

### 1. 一覧表示 (`list`)

リポジトリに関係するシンボリックリンクをテーブル形式で表示する。

表示項目:

| 種別 | リンクパス | リンク先 | 状態 |
| ------ | --------- | ------- | ------ |
| リポ内 | `./config/settings.yml` | `./config/settings.production.yml` | OK |
| 外部→リポ | `~/.config/myapp` | `~/workspace/ai-spec-starter/config` | OK |
| リポ内 | `./old_link` | `../deleted_dir/file` | リンク切れ |

- 種別は「リポ内」（リポジトリ内にあるリンク）と「外部→リポ」（外からリポジトリを指すリンク）
- リポジトリ内のパスは `./` からの相対パスで表示
- ホームディレクトリのパスは `~` に短縮
- 状態は「OK」または「リンク切れ」

### 2. リンク切れ検出 (`check`)

dangling symlink（リンク先が存在しないシンボリックリンク）を検出する。

```bash
# リポジトリ内のリンク切れ
find "$REPO_ROOT" -type l -not -path '*/.git/*' -not -path '*/node_modules/*' \
  -not -path '*/vendor/*' -not -path '*/.bundle/*' \
  ! -exec test -e {} \; -print 2>/dev/null
```

検出結果はリンク切れのみをフィルタして表示し、件数サマリーも添える。

### 3. シンボリックリンクの追加 (`add`)

新しいシンボリックリンクを作成する。

手順:

1. ユーザーからリンク元（作成するリンクのパス）とリンク先（実体ファイル）を確認
2. リンク先が実在するか確認。存在しない場合は警告して確認を取る
3. リンク元に既にファイル/リンクが存在する場合は上書き確認を取る
4. `ln -s <target> <link_path>` で作成
5. 作成後に `ls -la <link_path>` で結果を表示

リポジトリ内にリンクを作る場合は、可能な限り相対パスを使う（ポータビリティのため）。

### 4. シンボリックリンクの削除 (`remove`)

指定されたシンボリックリンクを削除する。

手順:

1. 対象がシンボリックリンクであることを `test -L` で確認（通常ファイルの誤削除を防止）
2. リンクの詳細（リンク先、状態）を表示してユーザーに確認
3. 確認後 `rm <link_path>` で削除
4. 削除結果を表示

一括削除（リンク切れをすべて削除など）の場合:

1. 対象リストを一覧表示
2. ユーザーに全件確認を取る
3. 1件ずつ削除し、結果をまとめて報告

### 5. リンク切れの修復 (`fix`)

リンク切れのシンボリックリンクに対して修復案を提示する。

修復戦略（優先順に試行）:

1. **候補検索**: リンク先のファイル名でリポジトリ内を検索し移動先を探す
2. **git log で追跡**: `git log --diff-filter=R --find-object` でリネーム履歴を確認
3. **類似パス推測**: リンク先パスのパターンから移動先を推測
4. **削除提案**: 修復不能な場合は削除を提案

修復の実行:

```bash
rm <link_path>
ln -s <new_target> <link_path>
```

### 6. 整理レポート生成 (`report`)

スキャン結果を Markdown レポートとして生成・保存する。

レポート構成:

```markdown
# シンボリックリンク整理レポート
リポジトリ: <REPO_ROOT>
生成日時: YYYY-MM-DD HH:MM

## サマリー
- リポジトリ内リンク: N 件（リンク切れ: N 件）
- 外部→リポジトリ: N 件（リンク切れ: N 件）

## リンク切れ一覧
| # | 種別 | リンクパス | リンク先（無効） | 推奨アクション |
|---|------|-----------|----------------|--------------|

## 正常なリンク一覧
| # | 種別 | リンクパス | リンク先 |
|---|------|-----------|---------|

## ディレクトリ別集計
| ディレクトリ | リンク数 | リンク切れ |
|------------|---------|----------|
```

デフォルト保存先: リポジトリルートの `symlink-report.md`（共通定義は `.claude/skills/paths.md` の **`SYMLINK_REPORT`** キーを参照）

### 7. gitignore チェック (`gitignore`)

「外部→リポ」のシンボリックリンクについて、リンク元リポジトリの `.gitignore` にリンク対象（例: `.claude`）が含まれているかを検証する。

シンボリックリンクが Git に追跡されると、リンク先の実体がリモートに露出するリスクがあるため、このチェックは重要。

#### チェック方法

```bash
# 外部→リポのシンボリックリンク元のリポジトリで .gitignore を確認
# リンクパスの basename（例: .claude）が .gitignore に含まれているか
LINK_NAME=$(basename "$link_path")
LINK_REPO=$(dirname "$link_path")
gitignore="$LINK_REPO/.gitignore"

if [ -f "$gitignore" ]; then
  # .gitignore 内にリンク名のパターンがあるか確認
  # 例: .claude, .claude/, /.claude のいずれかにマッチすれば OK
  grep -qE "^/?\.?${LINK_NAME}/?$" "$gitignore"
fi
```

#### 判定基準

- **OK**: `.gitignore` に `.claude`、`.claude/`、`/.claude` のいずれかが含まれている
- **部分的**: `.claude/settings.local.json` のようにサブパスのみ除外されている → `.claude` ディレクトリ自体の除外が必要
- **要追加**: `.gitignore` にリンク名が含まれていない、または `.gitignore` が存在しない

#### 出力形式

| # | リポジトリ | `.gitignore` のパターン | 状態 |
| --- | --- | --- | --- |
| 1 | techseek | `.claude/` `.claude` | OK |
| 2 | zenn-articles | なし | **要追加** |
| 3 | obsidian | `.claude/settings.local.json` のみ | **部分的** |

#### Git 追跡チェック

`.gitignore` の確認に加えて、リンク対象がすでに Git に追跡（tracked）されていないかも検証する。`.gitignore` に追加しても、すでに追跡されているファイルは無視されないため、このチェックは必須。

```bash
# リンク元リポジトリで .claude が追跡されているか確認
LINK_NAME=$(basename "$link_path")
LINK_REPO=$(dirname "$link_path")
tracked_count=$(git -C "$LINK_REPO" ls-files "$LINK_NAME" 2>/dev/null | wc -l | tr -d ' ')
```

#### 出力形式（統合）

| # | リポジトリ | `.gitignore` | Git 追跡 | 状態 |
| --- | --- | --- | --- | --- |
| 1 | techseek | `.claude/` | 0 件 | OK |
| 2 | zenn-articles | なし | 52 件 | **要修正** |
| 3 | obsidian | 部分的 | 554 件 | **要修正** |

#### 修正提案

状態が「要追加」「部分的」「要修正」の場合、以下を提案する:

1. **`.gitignore` へのエントリ追加**: `.claude` を1行追記（ファイル末尾に改行を確保して追加）。「部分的」の場合はサブパスのエントリを `.claude` に置換
2. **Git 追跡の解除**: `git rm -r --cached .claude` で追跡を解除（ワーキングツリーのファイルは削除されない）

いずれもユーザーの確認後に実行する。追跡解除後は各リポジトリで未コミットの変更が発生するため、コミットが必要な旨もあわせて案内する。

## ユーザーとのやりとり

- 操作を実行する前に必ず確認を取る
- 特に削除・修復は対象を明示してから確認
- 一覧が長い場合はディレクトリ別にグルーピングして見やすくする
- ユーザーが「整理して」と曖昧に言った場合は、まず `check`（リンク切れ検出）を実行し、結果を見せてから次のアクションを提案する
